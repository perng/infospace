Creating a Presentation in Infospace

A user's manual for authors — from a one-paragraph idea to a narrated journey through a 3D world.

What is Infospace?

Infospace replaces slides with a guided journey through a continuous 3D world. Your content — text, formulas, charts — lives at named places inside atmospheric environments: chalk on the boards of an evening lecture hall, formulas etched in glass slabs floating in a dark void, data as glowing holograms. The audience travels from stop to stop along a route you author, with cinematic camera moves, optional voice narration, and portals that dive between worlds.

The crucial idea for you as an author: you write meaning, the framework handles space. You never type a coordinate, place a camera, or pick a color. You say "this formula goes on the center board, framed close enough to read" — using names the worlds publish — and the layout solver, camera planner, and skins turn that into the finished scene. Because everything visual is derived from your semantic content, the same document also yields a readable outline, captions, and screen-reader text for free.

   │   npm run generate

Concepts in five minutes

TermWhat it is
WorldA 3D environment with its own atmosphere and scale — e.g. the marble atrium, the evening lecture-hall, the dark math-void. Worlds are prebuilt; you pick which ones your talk uses.
StationA named content slot a world publishes, like board.center or plinth.identity. Each station has a position and a size envelope — but you only ever use the name. One scene per station.
SceneOne stop on the tour: a piece of content, the station it lives at, how the camera frames it, and what the narrator says. Scenes play in the order you list them.
SkinThe 3D look of a piece of content: chalkboard renders a formula as chalk strokes that write themselves; hologram renders a bar chart as glowing volumetric bars. Every content kind has a sensible default skin.
Camera intentA named framing, like read-close (square-on, readable) or orbit-focus (three-quarter view for charts). The planner combines the intent with the station's size to compute the actual camera pose.
PortalA cinematic transition between worlds — the camera dives into something in one world and emerges in another (e.g. straight through the lecture-hall board into the math void).
BranchAn optional shortcut edge for live presenting — "skip to the reveal" — shown to the presenter without disturbing the main route.
NarrationA spoken script per scene. Stored as text (the source of truth); audio clips are rendered offline by a text-to-speech job, with synchronized captions.

Quick start: run an example

From the repository root:

npm install
cd examples/basel-problem
npm run dev

Open the local URL Vite prints (usually http://localhost:5173). You're standing in the lecture hall at the first board of "Basel Night Lecture" — a fully AI-generated talk. Press V to hear it narrate itself.

Three example presentations ship with the repo:

  • examples/svd-tour — the flagship: Singular Value Decomposition across the atrium and the spectral world, with rendered animations.
  • examples/math-primer — the minimal one: Euler's identity, lecture hall → math void.
  • examples/basel-problem — generated from a one-paragraph brief by the AI pipeline (Tutorial A follows its exact steps).

Three ways to create a presentation

PathYou writeBest for
A. Generate from a brief One paragraph of prose (brief.md) Getting a complete first draft in minutes, then editing. Recommended starting point.
B. Write the spec by hand A YAML file (talk.spec.yaml) — one block per scene Full control of every word, while still never touching coordinates.
C. The document layer TypeScript / JSON (the authored document) Advanced: images, video, rendered Manim animations, custom worlds. See Beyond the spec.

Paths A and B meet in the same file: generation produces a talk.spec.yaml that you then edit by hand. Everything in Tutorial B applies to a generated spec too.

Tutorial A — From a one-paragraph brief

This is exactly how examples/basel-problem was made. You write a paragraph; an AI engine — grounded in the framework's real vocabulary so it cannot invent places that don't exist — drafts the full talk; you review and refine it.

Step 1 — Set up a project

The easiest start is to copy the generated example and empty it:

cp -r examples/basel-problem examples/my-talk
cd examples/my-talk
rm -rf node_modules dist public/assets/narration talk.spec.yaml talk.provenance.json
# edit package.json: change "name" to "my-talk"
cd ../.. && npm install        # registers the new workspace

Two more things to personalize:

  • index.html — the page title.
  • src/main.tsx — which world components the app loads. It must match the worlds your talk uses:
<PresentationApp
  journey={journey}
  worlds={{ "lecture-hall": LectureHallWorld, "math-void": MathVoidWorld }}
/>

(Keep the lecture-hall + math-void pair for your first talk; the reference lists all four worlds.)

Step 2 — Write the brief

Replace brief.md with your own paragraph. Say what the talk is about, who it's for, roughly how many stops, the story arc, and the tone. The real Basel brief, for flavor:

# Brief

A guided talk on the **Basel problem** for second-year undergraduates,
about eight stops, suitable as a self-running narrated tour.

The story: Euler's 1734 solution of the Basel problem. Open by posing the
question — what is 1 + 1/4 + 1/9 + 1/16 + …? Show that the series
converges but creeps toward its limit hopelessly slowly (partial sums make
a nice chart). Walk through Euler's audacious move: treating sin(x)/x as
an infinite polynomial and reading off its roots, comparing coefficients
to conjure the answer. The reveal — pi squared over six — should land as
the dramatic high point somewhere otherworldly. Close with why it
mattered: the birth of the zeta function and the road toward Riemann.

Tone: wonder with rigor — a night lecture that ends somewhere the
blackboard cannot go. Use the lecture hall for the setup and the series
work, then pass through the board into the void for Euler's leap and the
reveal.
What makes a good brief Name the dramatic high point and where you want it to land ("somewhere otherworldly"); mention data that wants to be a chart; describe the tone. The generator maps these cues onto worlds, skins, and camera framings.

Step 3 — Generate

npm run generate

This runs the generator with your brief and the worlds you chose (see the generate script in package.json — change --worlds lecture-hall,math-void there if you picked different worlds). It calls a locally installed AI CLI: the claude CLI by default, or pass --engine codex for the Codex CLI. Either must be installed and logged in.

Every draft is validated through the full compiler before it's accepted — invalid output gets one automatic repair round, then is rejected. A successful run writes three files:

  • talk.spec.yaml — the Authoring Spec: the talk as editable YAML. This is now your source file.
  • src/journey/document.json — the compiled document the app loads.
  • talk.provenance.json — a record of the brief, engine, model, and attempts.

Run npm run dev and take the tour.

Step 4 — Review and edit

Generation gives you a structurally valid draft; judgment is still your job. Walk the tour and read talk.spec.yaml critically. When the Basel talk was generated, the human pass made exactly two edits — both one-liners in the spec:

  • A camera intent that didn't suit its content (changed camera: sky-gaze to read-close on a formula that needed to be read).
  • An opening paragraph too long for its board (trimmed the body: text).

After any edit, recompile and check:

npm run spec:compile
npm run validate
npm run dev

Step 5 — Lock what's good, regenerate the rest

When a scene is exactly right, mark it:

  - id: reveal-answer
    locked: true
    title: Pi Squared Over Six
    ...

Then npm run generate:regen rewrites the talk from the (possibly updated) brief while preserving every locked scene byte for byte. This is the editing loop: keep the moments that work, let the AI rework the connective tissue around them.

Tutorial B — Writing the spec by hand

The Authoring Spec is a YAML file: a header, a list of scenes, and optional portals and branches. You only ever use names — station names, camera intent names, skin names — from the reference tables below.

Anatomy of a scene

  - id: definition          # kebab-case, unique within the talk
    title: Defining Phi     # shown in the outline, palette, minimap
    world: lecture-hall     # a world template id (listed in `worlds:`)
    station: board.center   # a station that world publishes — one scene per station
    role: detail            # chapter | detail | chart | quote | media | portal
    content:                # what this stop says (see Content kinds)
      kind: formula
      latex: \varphi = \frac{1 + \sqrt{5}}{2}
      spoken: phi equals one plus the square root of five, all over two
    skin: chalkboard        # optional — omit for the content kind's default
    camera: read-close      # optional — a camera intent (default read-close)
    narrate: >              # optional — the spoken script for the narrated tour
      Solving the proportion gives a quadratic equation, and its
      positive root is phi.
    notes: Derive it live if the room wants the algebra.   # presenter notes
    arrive: fly             # optional — how the camera travels here: cut | dolly | fly
    locked: false           # optional — survives --regen verbatim when true

Only id, title, world, station, and content are required. The defaults are good: omit skin and camera unless you want a specific effect.

A complete talk: "The Golden Ratio"

A six-scene talk using the same two worlds as the Basel example — boards in the lecture hall for the setup, then a portal through the board into the void for the payoff. This file compiles as-is:

id: golden-ratio
title: The Golden Ratio - a short walk
worlds:
  - lecture-hall
  - math-void
scenes:
  - id: opening
    title: An Old Obsession
    world: lecture-hall
    station: board.left
    role: chapter
    content:
      kind: text
      title: The golden ratio
      body: |
        Cut a line so the whole relates to the long part
        as the long part relates to the short.

        That single sentence defines phi.
    skin: chalkboard
    narrate: >
      For more than two thousand years, one proportion has fascinated
      geometers, painters, and architects. Tonight we meet it properly.

  - id: definition
    title: Defining Phi
    world: lecture-hall
    station: board.center
    content:
      kind: formula
      latex: \varphi = \frac{1 + \sqrt{5}}{2} \approx 1.618
      spoken: phi equals one plus the square root of five, all over
        two, approximately one point six one eight
    narrate: >
      Solving the defining proportion gives a quadratic equation, and
      its positive root is phi.

  - id: fibonacci
    title: Fibonacci Closes In
    world: lecture-hall
    station: board.right
    role: chart
    content:
      kind: chart
      data:
        type: bar
        title: Ratios of consecutive Fibonacci numbers
        series:
          - { label: 3/2,    value: 1.5 }
          - { label: 8/5,    value: 1.6 }
          - { label: 21/13,  value: 1.6154 }
          - { label: 55/34,  value: 1.6176 }
          - { label: 144/89, value: 1.618 }
      summary: Ratios of consecutive Fibonacci numbers settle toward
        one point six one eight.
    camera: orbit-focus
    narrate: >
      The Fibonacci numbers sneak up on phi. Divide each by the one
      before it, and the ratios settle onto the same constant.

  - id: leaving-the-hall
    title: Through the Board
    world: lecture-hall
    station: wall.west
    role: portal
    content:
      kind: text
      title: A self-similar number
      body: |
        Phi is the number that contains itself.

        To see that, we leave the hall.
    skin: cloudText
    camera: wide-establishing
    narrate: >
      What makes phi special is not its decimals but its self-similarity.
      To see it, we step through the board.

  - id: self-similar
    title: Phi Squared
    world: math-void
    station: plinth.identity
    content:
      kind: formula
      latex: \varphi^{2} = \varphi + 1
      spoken: phi squared equals phi plus one
    skin: etchedGlass
    camera: quote-gaze
    narrate: >
      Square phi and you get phi back, plus one. No other positive
      number does this. Every famous property of the golden ratio
      unfolds from this single identity.

  - id: closing
    title: Everywhere and Nowhere
    world: math-void
    station: circle.overlook
    content:
      kind: text
      title: Where phi lives
      body: |
        Pentagons. Sunflowers. Continued fractions.

        phi = 1 + 1/(1 + 1/(1 + ...))
    skin: cloudText
    camera: quote-gaze
    narrate: >
      Phi appears wherever growth repeats its own rule. That is where
      we leave it, glowing over the void.

portals:
  - from: leaving-the-hall
    to: self-similar
    label: Step through the board
    dive: board-dive          # a pose of the source world (lecture-hall)
    emerge: void-emerge       # a pose of the target world (math-void)

branches:
  - { from: opening, to: fibonacci, label: Skip to the evidence, kind: branch }

Things worth noticing:

  • Scene order is the route. The tour visits scenes in list order; you never wire up edges for the main path.
  • The portal is mandatory there. leaving-the-hall and self-similar are consecutive but in different worlds — the compiler refuses to guess a world crossing and demands a portal with a dive/emerge pose pair.
  • Skins follow content by default. The definition scene omits skin:, so its formula gets the chalkboard. The void scenes opt into etchedGlass and cloudText for atmosphere.
  • In-world text is plain ASCII. The 3D fonts have no Greek or subscripts. Write "phi", "n squared", "1/n^2" in text bodies and titles; real notation belongs in latex:, which renders through a full math pipeline.
  • Every formula and chart carries its spoken form (spoken: / summary:) — that's what the outline, captions, and screen readers use.

Compile and iterate

npm run spec:compile     # talk.spec.yaml → src/journey/document.json
npm run validate         # full structural + coverage report
npm run outline          # the talk as a readable text outline
npm run dev              # walk it

validate reports the resolution summary — how many cameras came from intents, route coverage, narration coverage — and any errors with the exact names that failed. outline is the fastest way to proofread the words without launching the 3D view.

Sizing text to a station Boards in the lecture hall are about 3.8 × 2.8 units; a text body of roughly three short paragraphs fits. If text looks cramped in the 3D view, cut words — don't fight the layout. Bigger stations (circle.overlook, arrival.overlook) take more.

Voice narration

Narration is authored as text — the narrate: field on each scene — and rendered to audio offline. The script text is the source of truth; audio is a derived, cached asset.

npm run narration            # print the scripts as JSON (proofread pass)
npm run narration:render     # render audio clips with local TTS

Rendering uses the Kokoro text-to-speech engine locally (model files required; see packages/cli/generate-narration.py). Clips land in public/assets/narration/ with a manifest of word timestamps — that's what powers the synchronized karaoke captions. Clips are hash-cached: only scenes whose script changed are re-rendered.

In the viewer, V toggles the narrated tour: each stop plays its clip with captions and auto-advances when it ends. Write narrate: in a spoken style — two to four sentences, numbers and symbols written out as words ("pi squared over six", not "π²/6").

Validation knows about staleness npm run validate warns when a script changed after its clip was rendered, so you can't accidentally ship narration that says something different from the document.

Camera intents

IntentFraming
read-closeSquare-on, close enough to read body text comfortably. The default.
wide-establishingSteps back and looks past the content into the world behind it — good for arrivals and departures
orbit-focusOff-axis three-quarter view — flatters charts and 3D objects
sky-gazeFrom low ground, gazing up at content overhead — for very high stations
quote-gazeSlightly below floating text, looking up at it

You pick the intent; the planner computes the actual camera position from the station's size and facing. A scene's camera: may also name one of a world's fixed poses (e.g. prism-approach) for framings about the world's own geometry.

Content kinds (in the spec)

KindFieldsNotes
texttitle (optional), bodyPlain ASCII. About three short paragraphs fits a board.
formulalatex, spokenPresentation-sized LaTeX — one line or a small aligned block. spoken is the spoken-math reading.
chartdata, summarydata.type is bar (title, optional unit/logScale, series of label/value) or scatter (title, xLabel, yLabel, points of x/y/label). Small inline datasets only. summary is a one-sentence reading.

Images, video, and Manim animations need asset files on disk, so they enter at the document layer, not the spec.

LaTeX in YAML Write latex: as an unquoted (plain) YAML scalar — backslashes pass through as-is, e.g. latex: \frac{\pi^{2}}{6}. If you must quote with double quotes, every backslash doubles: "\\frac{\\pi^{2}}{6}". Unquoted is easier.

Portals and branches

portals:                      # REQUIRED between consecutive scenes in different worlds
  - from: leaving-the-hall    # scene id in the source world
    to: self-similar          # scene id in the target world
    label: Step through the board
    dive: board-dive          # pose name from the source world (optional but recommended)
    emerge: void-emerge       # pose name from the target world

branches:                     # optional shortcuts for live presenting
  - from: opening
    to: fibonacci
    label: Skip to the evidence
    kind: branch              # branch (skip ahead) | return (go back)

Portals are the only way the tour changes worlds; the dive/emerge poses choreograph the crossing (accelerate into the board, plunge out of the sky). Branches never interrupt the main route — they appear as labeled options in the presenter console and on the minimap.

Beyond the spec: the document layer

The YAML spec deliberately covers only content that needs no external assets, so a talk always renders. For more, edit the authored document itself (the layer below the spec):

  • Images and video — plaque-framed media referencing files in public/.
  • Manim animations — write a Manim scene in Python; the render job produces a transparent clip with cuepoints, and the projection skin plays it composited over the world. With reveal: stepwise, each press of → advances one animation step, and narration cue marks drive the same steps in the self-running tour.
  • Hand-tuned framings — explicit camera poses where an intent isn't enough.

examples/svd-tour shows all of this in action (see its src/journey/project.ts). The document is still positionless — stations and intents — it just speaks the framework's full vocabulary. The complete data model lives in docs/design.md, and docs/creation-guide.md is a condensed reference (written for AI assistants, equally useful to humans who want every field on one page).

Troubleshooting

Symptom / messageFix
…are consecutive but in different worlds — add a portal… Two back-to-back scenes change worlds without a portal. Add an entry to portals: from the first to the second, with a dive/emerge pose pair.
station "…" is double-booked by anchors… Two scenes claim the same station. Move one to a free station (each world's table above is the menu).
…references unknown station "…" (available: …) Typo, or the station belongs to a different world. The error lists the valid names.
unknown world template "…" worlds: must use template ids exactly: atrium, spectral, lecture-hall, math-void.
Formula renders blank or wrong Check the YAML escaping (see the LaTeX callout), and keep to presentation-sized standard LaTeX (amsmath); exotic packages aren't available.
Text spills over its board The station is smaller than your paragraph. Cut words or move to a bigger station — boards are 3.8 × 2.8.
Greek letters / symbols missing in 3D text In-world text is ASCII-only. Spell it out ("pi", "n squared"); use a formula scene for real notation.
Narration sounds stale after editing Re-run npm run narration:render — only changed scripts are re-rendered. npm run validate lists stale clips.
npm run generate fails to start an engine The default engine is the claude CLI and it must be logged in; otherwise install/log in the codex CLI and add --engine codex to the generate script.
Edits to talk.spec.yaml don't show up The app loads document.json, not the YAML. Run npm run spec:compile after every spec edit.
